# コントリビューションガイド

この研修では「既に動いているコードに、チーム開発の作法を守って手を入れる」練習をします。
以下のルールに沿って作業してください。

## ブランチ命名規則

`main` には直接コミットせず、必ずブランチを切ってから作業します。

| 種別 | プレフィックス | 例 |
|---|---|---|
| 機能追加 | `feature/` | `feature/order-history` |
| バグ修正 | `fix/` | `fix/coupon-discount` |
| リファクタ | `refactor/` | `refactor/order-service` |
| ドキュメント | `docs/` | `docs/readme-setup` |

## コミットの作法

- 1 コミット＝1 つの意味のある変更にまとめる。
- コミットメッセージは「何をしたか」が一目で分かる日本語または英語で書く。
  - 例: `クーポン割引の計算を税抜小計ベースに修正`
  - 例: `fix: restore stock on order cancel`
- 動かない状態（ビルドが通らない・テストが赤い）でコミットしない。

## プルリクエストの作法

1. ブランチを push し、`main` に向けて PR を作成する。
2. PR テンプレート（`.github/pull_request_template.md`）の各項目を埋める。
   - 何を変えたか / どう確認したか / 影響範囲
3. CI（`dotnet test`）がグリーンであることを確認する。**赤い PR はレビューに出さない。**
4. レビュー指摘には、修正コミットを追加して対応する（履歴を消さない）。

## 動作確認の基本

```bash
dotnet test   # すべてグリーンであること
dotnet run --project src   # 起動して Swagger UI で API を叩いて確認
```

「テストがグリーン＝壊していない目印」です。直したら必ずテストを走らせてください。
